# Medusa Ransomware — Detection Engineering Project

A detection-engineering workflow exercise: taking a real ransomware actor's signature behaviors, verifying them against authoritative threat intelligence, authoring detections, and validating them end-to-end in a lab.

## What this project demonstrates

- Reading adversary TTPs from a CISA advisory and reconciling them against MITRE ATT&CK's live objects (Medusa Group **G1051**, Medusa Ransomware **S1244**).
- Accounting for ATT&CK versioning drift (the v19 Defense Evasion restructure — e.g. `T1562.001` → `T1685`).
- Authoring vendor-neutral **Sigma** rules and compiling them to **Splunk SPL** with `sigma-cli`.
- Validating a detection against real attacker telemetry generated by **Atomic Red Team**, including proving both true-positive firing and negative-case specificity.
- Diagnosing a real pipeline/field-mapping mismatch (`EventID` vs `EventCode`) between a converter's output and actual SIEM ingest.

## Honest scope statement

This repo is a **workflow and lab-validation exercise**, not a claim of detection-engineering mastery. The distinction is deliberate and maintained throughout:

- **Authored and understood end-to-end:** the T1490 vssadmin shadow-copy-deletion rule. This rule was hand-designed, its design decisions can be explained (dual binary-identity match, token-not-flag matching, deliberate scope), and it was validated in the lab with documented positive and negative tests.
- **Generated for coverage:** the remaining rules were produced from the verified target list and compiled with `sigma-cli`. They are technically correct and traceable to authoritative ATT&CK mappings, but were not hand-authored rep-by-rep. They are included so the coverage map is complete, and are labeled as such.

## Environment

Windows VM with Sysmon (process creation, Event ID 1) → log forwarder → Splunk. Atomic Red Team for attacker-behavior simulation. Detections authored in Sigma, compiled to SPL with `sigma-cli` (splunk backend, `splunk_windows` and `sysmon` pipelines).

## Detection coverage

Target behaviors, verified against G1051 / S1244 (see `coverage-map.md` for full verification detail):

| # | Behavior | Technique | Status |
|---|----------|-----------|--------|
| 1 | Shadow copy deletion (`vssadmin delete shadows`) | T1490 | Authored + lab-validated |
| 2 | Service Stop (`taskkill /F /IM`, `net stop`) | T1489 | Generated + lab-validated |
| 3 | PowerShell history wipe | T1070.003 + T1690 | Generated + lab-validated |
| 4 | Encoded PowerShell | T1059.001 + T1027.010 | Generated (not lab-validated) |
| 5 | RDP enablement (registry + firewall) | T1112 + T1686 | Generated (process-creation proxy; not lab-validated) |
| 6 | BYOVD — disable security tools | T1685 (+T1543.003, T1553.002) | Generated (not lab-validated — see rule scope note) |

**Validation summary:** 3 of 6 lab-validated in Splunk against Atomic Red Team / direct
attacker-command stimulus (T1490, T1489, T1070.003). The remaining 3 are authored from the
verified MITRE mapping but not lab-validated: encoded PowerShell was scoped out for time,
and RDP enablement and BYOVD are documented as needing non-process-creation telemetry
(registry_set / driver_load events) plus YARA to validate properly. This split is deliberate —
forcing clean-looking passes on techniques that don't validate in an afternoon (notably BYOVD)
would misrepresent the work.

## Key finding (T1490 validation)

The rule fires on the *execution attempt*, independent of whether shadow copies existed to delete — the correct behavior for a pre-encryption ransomware canary. Specificity was proven empirically: over the same test window, the detection query returned only the `delete shadows` event while a broad query returned both the `delete` and the benign `list shadows` events, confirming the rule discriminates on command behavior, not just on the vssadmin binary. See `test-evidence/T1490/`.

## Notes and honest limitations

- Rules are lab-validated, not production-tested; false-positive surfaces are documented, not tuned against real-world baselines.
- `taskkill`/`net stop` (T1489) have a large benign-admin false-positive surface; a production version would need correlation against security/backup service-name targets or a suspicious-parent signal, which is documented as a tightening approach rather than implemented.
- The `EventID` vs `EventCode` field mismatch encountered during conversion is a real-environment lesson: a converter's field assumptions must be reconciled against how the SIEM actually ingests data before deployment.
